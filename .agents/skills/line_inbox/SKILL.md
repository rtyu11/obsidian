# LINE Inbox Processor Skill

## トリガー条件
- **「LINEを処理して」** または **「LINEを整形して」**：`LINE/` 内の未処理ファイルを分類し、各フォルダへ振り分ける

---

## フォルダ構成

```
LINE/                      ← Make.com が同期するテキストメモ (.md)
LINE/images/               ← Make.com が同期する画像ファイル
LINE/pdfs/                 ← LINEから送付されたPDF（存在する場合のみ）
LINE/audio/                ← 音声ファイル（処理対象外・手動転記要）

Ideas/            ← アイデア・企画・着想
  アイデアリスト.md
  [個別ファイル].md
Tasks/            ← 課題・やること
  課題リスト.md
  [個別ファイル].md
Memo/             ← 雑多メモ・気づき
  メモリスト.md
  [個別ファイル].md
Quotes/           ← 名言・格言
  名言集.md         ← 常にここに追記（独立ファイルは作らない）
Journal/          ← 日記・エッセイ・観察・空想・自分の考えや感情が入る文章
  [個別ファイル].md
Source/OCR/       ← 本ページ・手書きの全文OCR保存先
Notes/Books/      ← 本ごとのノート
Insights/         ← 深い洞察・統合的な思考の記録
```

---

## 実行手順

### 1. 対象ファイルの収集とグループ化

以下をそれぞれ確認する。

**テキストメモ（`LINE/*.md`）:**
- `LINE/` 直下の `.md` ファイルを一覧取得
- `_processed_` プレフィックスのファイルはスキップ

**画像（`LINE/images/`）:**
- `LINE/images/` 内のファイルを一覧取得
- `_processed_` プレフィックスのファイルはスキップ

**PDF（`LINE/pdfs/`）:**
- フォルダが存在する場合のみ確認
- `_processed_` プレフィックスのファイルはスキップ

**音声ファイル（`LINE/audio/`）:**
- `LINE/audio/` 内のファイルを一覧取得（`.m4a` `.mp3` `.ogg` `.aac` `.wav` `.flac` `.opus`）
- `_processed_` プレフィックスのファイルはスキップ
- 未処理の音声ファイルが存在する場合 → 手順 4（音声ファイルの処理）へ進む
- 音声ファイルが存在しない、またはすべて処理済みの場合はスキップ

すべてのソースが空の場合は「LINEフォルダに未処理のファイルはありませんでした」と伝えて終了する。

#### 1-1. ファイルのグループ化

収集したすべてのファイルを時系列（ctime）に並べ、以下の2つの軸でグループ化する。

**① 時間軸グループ（前後2分以内）**

テキスト・画像・音声のうち作成時刻が前後2分以内のファイルを同一グループとして扱う。

用途：テキストの補足説明を直後の画像・音声に引き継ぐ。

**② 本タイトル軸グループ（時間を超える）**

画像を開いて本のページと判断した場合、そのタイトルで同定し、**時間が離れていても同一タイトルの画像・気づきメモはすべて同じ本のグループ**として扱う。

- 本ページ画像 → タイトルを特定して `Notes/Books/[タイトル].md` に紐づける
- 本ページ画像の前後にある気づきテキスト・音声（「この部分が面白い」「ここから気づき」など）→ 同じ本のノートに追記する
- 間に別のテキストやメモが挟まっていても、内容が明らかにその本についての気づきなら本のグループとして扱う
- 本と無関係と判断したテキスト・画像は別グループとして独立処理する

**ユーザーの運用ルール（前提として把握すること）：**
- **同じ本の続き** → ユーザーは何も送らない。ファイルのタイムスタンプと内容の連続性で判断する。迷ったら確認する
- **本が変わるとき** → ユーザーが「ここから〇〇」などのテキストを送る。それを境界として扱う
- **本についての気づきメモ・音声** → 本ページ画像と時間が離れていても、内容から同じ本と判断できれば `Notes/Books/[タイトル].md` に追記する
- タイトルが判断できない場合のみ「この画像はどの本ですか？」と確認する

**グループ処理の共通ルール：**
- テキストに種別指定（`#ideas` 等）や文脈説明がある → グループ内の画像・音声にもその分類・文脈を適用する
- **テキスト単体で意味をなさない内容**（「これ参照」「↓」「画像の通り」「この本いい」など補足的な短文）→ 直後の画像・音声に紐づけて処理し、テキスト単独では保存しない
- テキストなし・画像のみ → 画像の内容から単独で分類する
- グループ内に複数の画像がある場合 → 各画像を個別に処理するが、テキストの文脈は全画像に適用する

### 2. テキストメモの処理（`LINE/*.md`）

#### 2-1. 日付の取得

Make.com が生成したファイル名（例: `memo_YYYYMMDD_HHmmss.md` 等）から日付を取得する。
またはファイルの作成日時・更新日時を使用する。

#### 2-2. コンテンツ種別の分類

本文の内容からコンテンツ種別を判断する。

**ラベル指定がある場合（優先）:**

| タグ | 振り分け先 |
|------|-----------|
| `#ideas` / `#idea` | `Ideas/` |
| `#tasks` / `#task` | `Tasks/` |
| `#memo` | `Memo/` |
| `#quotes` / `#quote` | `Quotes/名言集.md` |

**ラベルなし・自動判定:**

| 内容の性質 | 振り分け先 |
|-----------|-----------|
| 新しいアイデア・企画・「やってみたい」 | `Ideas/` |
| 解決したい課題・やること・TODO | `Tasks/` |
| 雑多なメモ・気づき・知識メモ（「私」が主語でない）、短い自己観察・一言の気づき・問い | `Memo/` |
| **一言の名言っぽい言葉**（自分の言葉でも格言的・普遍的な表現になっているもの） | `Quotes/名言集.md` |
| 他者の言葉・名言・格言（出典が他者） | `Quotes/名言集.md` |
| 本の内容についての読書メモ | `Notes/Books/[タイトル].md` |
| 深い考察・複数テーマにまたがる洞察 | `Insights/` |
| 日記・エッセイ・日常の観察・空想・自分の考えや感情が入る文章（「私」が主語）、かつ**ある程度の文章量がある（複数文以上）** | `Journal/` |

> **Journalの判定基準（重要）**：`Journal/` は文章量があるもの限定。一言・短文の観察や気づき・問いは `Memo/` に入れる。名言っぽい一言は `Quotes/名言集.md` に入れる。`Memo/` か `Journal/` か迷う場合は**文章量が複数文あるか**で判断する。

**判断が曖昧な場合:** 最も近い複数のフォルダに同時追記する（例：アイデアでもありタスクでもある → 両方）。

**本文が空のファイル:** 振り分けせずそのまま削除する。

**判断が全くできない場合:** ユーザーに「このメモ（内容の冒頭）はアイデア・タスク・メモのどれですか？」と確認する。

**名言は常に `Quotes/名言集.md` の末尾に追記する（独立ファイルは作らない）。**

#### 2-3. 軽いメモ vs 独立ファイルの判定

**リスト型追記の条件（以下をすべて満たすもの）:**
- 本文（タイトル・空行・タグ行を除いた実質的な内容）が **3行以下**
- 見出し（`# ##`）や番号付きリスト（`1. 2.`）などの **構造化マークダウンがない**

→ 振り分け先リストファイルの末尾に追記する

**独立ファイル型の条件（以下のいずれかを満たすもの）:**
- 本文が **4行以上**
- 見出しまたは番号付きリストなど **構造化マークダウンがある**
- 名前付きの概念・トピックとして単体で価値を持つ内容

→ 振り分け先フォルダに独立ファイルを作成する

#### 2-4. 追記・保存フォーマット

**リスト型追記フォーマット:**
```markdown
### YYYY-MM-DD
- [内容]
```

本文が複数行の場合は箇条書きで展開する：
```markdown
### YYYY-MM-DD
- [1行目の内容]
- [2行目の内容]
```

整形ルール：
- 箇条書き入力は適切な書き言葉に直す（意味は変えない）
- タグ（`#ideas` 等）は削除する

**独立ファイル型フォーマット:**

frontmatter（必須）：
```yaml
---
date: YYYY-MM-DD
tags: [カテゴリタグ, トピックタグ1, トピックタグ2]
source: line
---
```

カテゴリタグ（1つ付与）：
- Ideas → `ideas`
- Tasks → `tasks`
- Memo → `memo`

トピックタグ（内容のキーワードから2〜4個判断して付与）：内容に合わせて適宜付与

本文は元の内容を構造を保ったまま使用する。タグ行（`#memo` 等）は削除する。

**Obsidianリンク・タグの付与（必須）：**

保存時に以下を本文末尾に付与する：

```markdown
---
関連：[[既存ノート名]] [[既存ノート名]]
#日本語タグ1 #日本語タグ2
```

- `[[リンク]]`：Vault内の既存ノート（`Notes/Books/`・`Notes/Topics/` 等）で関連するものがあればリンクする。なければ省略。
- `#日本語タグ`：内容のキーワードを日本語タグで2〜4個付与する（例：`#習慣と継続` `#アイデア` `#脳科学`）。frontmatterのtagsとは別に本文末尾にも付与する。

**リスト型追記の場合も**、同日の日付ブロック末尾に関連リンクがあれば追記する。ただし軽いメモには不要。

#### 2-5. 処理後の削除

振り分け完了後、`LINE/` の元 `.md` ファイルを削除する。

---

### 3. 画像ファイルの処理（`LINE/images/`）

画像をビジョンで確認し、内容に応じて処理方法を分岐する。同グループのテキストがある場合はその文脈（種別・説明）を優先して適用する。

| 画像の内容 | 処理方法 |
|-----------|---------|
| 本のページ（印刷文字・手書き書き込みあり） | 手順 3-1（book_highlighter フロー）へ |
| 手書きメモ・ノート・走り書き | 手順 3-2（手書きフロー）へ |
| 記事・Web・アプリのスクリーンショット | 手順 3-3（スクショフロー）へ |
| その他（判断できない）| ユーザーに確認する |

#### 3-1. 本のページ画像（book_highlighter フロー）

`book_highlighter` SKILL.md の手順 2〜5 をそのまま適用する：
1. 本のタイトルを判定する（同グループのテキストにタイトルが書かれていればそれを使う）
2. 全テキストを `Source/OCR/[タイトル].md` に保存する
3. 色ペン（赤★・青※・緑＋）の書き込みを転送先に振り分ける
4. 処理済み画像を `_processed_` プレフィックスでリネームする

**タイトル判定の優先順位：**
1. 同グループのテキストにタイトルが明記されている → それを使う
2. 画像内に著者名・タイトル・柱（ページ端のタイトル）が写っている → そこから判定
3. 直前に処理した本のページ画像と文体・レイアウト・内容が明らかに連続している → 同一タイトルとして扱う
4. 上記いずれも判断できない → **「この画像はどの本ですか？」とユーザーに確認してから処理する**（推測で別タイトルとして保存しない）

#### 3-2. 手書きメモ画像（手書きフロー）

1. 画像からテキストをOCRで読み取る
2. 同グループのテキストに種別・文脈があればそれを適用、なければ内容から判断する
3. 手順 2-2 の分類基準に従い振り分け先を決定する
4. 独立ファイルとして保存する（手書きメモは通常4行以上）
5. 処理済み画像を `_processed_` プレフィックスでリネームする

#### 3-3. スクリーンショット・記事画像フロー

1. 画像からテキストを読み取り、何についての情報かを把握する（記事・検索結果・アプリ画面など）
2. 同グループのテキストに「これはメモ」「アイデア」などの文脈があればそれを優先して分類
3. なければ内容から手順 2-2 の基準で分類する
4. 振り分け先に保存する（通常は独立ファイル型。引用・参照として扱う）
5. 処理済み画像を `_processed_` プレフィックスでリネームする

### 4. 音声ファイルの処理（`LINE/audio/`）

Claude Code は音声ファイルを直接処理できないため、Gemini API スクリプトを使う。

#### 4-1. 前提確認

以下を確認する：
- `GEMINI_API_KEY` 環境変数が設定されているか
- Python と `google-genai` パッケージがインストールされているか（`pip show google-genai` で確認）

未設定の場合は以下を案内して処理をスキップする：
```
音声ファイルが N 件あります。処理するには:
1. https://aistudio.google.com でGemini APIキーを取得
2. 環境変数 GEMINI_API_KEY にセット（setx GEMINI_API_KEY "キー" でターミナル再起動後に有効）
3. pip install google-genai==1.16.0
4. python .agents/scripts/gemini_transcribe.py を実行
5. 再度「LINEを処理して」と指示してください
```

#### 4-2. スクリプト実行

```bash
python .agents/scripts/gemini_transcribe.py
```

スクリプトは以下を行う：
- `LINE/audio/` 内の未処理音声ファイルを Gemini 3.1 Flash Lite で文字起こし
- 話し言葉 → 書き言葉に整形
- 「これは名言です」など種別の発言から自動でタグを付与（`#quotes` / `#ideas` / `#tasks` / `#memo`）
- `LINE/YYYY-MM-DD_audio_[元ファイル名].md` として保存
- 元の音声ファイルを `_processed_` リネーム

#### 4-3. 生成された .md ファイルの処理

スクリプト実行後、`LINE/` に出力された `.md` ファイルを **手順 2（テキストメモの処理）** と同じ手順で分類・振り分けする。
frontmatter の `source: line-audio` を見て音声由来と識別できる。

---

### 5. PDFファイルの処理（`LINE/pdfs/`）

フォルダが存在する場合、`scribe_inbox` SKILL.md の手順に従い処理する。
処理後は `_processed_` プレフィックスを付けてリネームする（削除しない）。

【画像】

## 音声ファイルについて

Claude Code は音声ファイルを直接処理できない。`.agents/scripts/gemini_transcribe.py` を使い Gemini API 経由で文字起こしする。
セットアップが未完了の場合は手順 4-1 の案内を表示してスキップする。

---

## 禁止事項
- `Source/` 配下のファイルは編集・移動・削除すべて禁止
- 新規フォルダの作成禁止
- frontmatterに定義したタグ以外のインラインタグを勝手につけない
- すべての返答・ノートは**日本語**で行う
